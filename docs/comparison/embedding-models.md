# Embedding 模型对比

本文对比主流 Embedding 模型在 Delphi 场景下的适用性，重点关注离线部署能力与代码搜索效果。

## 输出类型对比：Dense vs Sparse vs ColBERT

不同模型支持不同的向量输出类型，直接影响检索策略的选择。

**Dense（稠密向量）**
- 将文本压缩为单一固定维度向量
- 适合语义相似度搜索，召回率高
- 无法精确匹配关键词，对专有名词（如函数名、API 名）不敏感
- 几乎所有模型都支持

**Sparse（稀疏向量）**
- 类似 BM25，保留词频权重信息
- 对精确关键词匹配效果好，适合代码中的标识符搜索
- 与 Dense 互补，混合使用效果最佳
- 仅 BGE-M3、SPLADE 等少数模型原生支持

**ColBERT（多向量交互）**
- 为每个 token 生成独立向量，检索时做 late interaction
- 精度最高，但存储开销大（每文档存储多个向量）
- 适合高精度重排序场景
- BGE-M3 是少数同时支持三种输出的模型之一

BGE-M3 的多输出能力使其可以在单次编码中同时产出 Dense、Sparse、ColBERT 三种表示，为混合检索提供原生支持。

## 离线部署可用性

Delphi 的核心设计目标是**完全离线运行**，这一约束直接淘汰了部分模型：

| 模型 | 离线可用 | 原因 |
|------|---------|------|
| BGE-M3 | 可用 | HuggingFace 开源，可本地加载 |
| OpenAI text-embedding-3 | 不可用 | 仅 API 访问，需联网 |
| Jina Embeddings v3 | 可用 | 开源权重，支持本地推理 |
| E5-Mistral-7B | 可用（受限）| 需要 7B 参数量的 GPU，资源消耗大 |
| Cohere Embed v3 | 不可用 | 仅 API 访问，需联网 |
| GTE-Qwen2-7B | 可用（受限）| 同样需要大显存 GPU |

闭源 API 模型（OpenAI、Cohere）在离线场景下完全不可用，直接排除。

## 代码搜索场景的特殊考量

代码搜索与自然语言搜索存在本质差异：

- **标识符精确匹配**：函数名、变量名、类名需要精确匹配，Dense 向量容易将语义相近但名称不同的符号混淆
- **跨语言语义**：同一逻辑在不同编程语言中表达不同，需要模型理解代码语义而非仅匹配字面
- **长上下文**：完整函数体可能超过 512 token，需要模型支持足够长的输入
- **注释与代码混合**：文档注释、行内注释与代码逻辑混合，模型需同时理解两者

BGE-M3 在 CodeSearchNet 等代码检索基准上表现优异，其 Sparse 输出对标识符的精确匹配尤为有效。

## 为什么 Delphi 选择 BGE-M3

综合以上分析，BGE-M3 是 Delphi 的最优选择：

1. **离线可用**：完整开源，权重可本地加载，无需任何网络连接
2. **多输出类型**：单模型同时支持 Dense + Sparse + ColBERT，天然支持混合检索，无需部署多个模型
3. **长文本支持**：8192 token 的输入长度足以覆盖大多数函数和类定义
4. **多语言能力**：支持 100+ 语言，中英文代码注释混合场景表现稳定
5. **资源友好**：相比 E5-Mistral-7B 和 GTE-Qwen2-7B，BGE-M3（约 570M 参数）在 CPU 上也可运行，降低了硬件门槛
6. **社区活跃**：由 BAAI 维护，持续更新，文档完善，HuggingFace 集成成熟

对于需要在开发者本地机器上运行的 Delphi 而言，BGE-M3 在精度、资源消耗和部署便利性之间取得了最佳平衡。

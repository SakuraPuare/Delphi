# 上下文窗口 (Context Window)

上下文窗口是 LLM 单次推理能处理的最大 token 数量。它直接决定了 RAG 系统能向模型提供多少参考信息，是影响回答质量的关键因素之一。

## 什么是上下文窗口

可以把上下文窗口理解为模型的"工作记忆"：

```
┌─────────────────────────────────────────────────────┐
│                   上下文窗口（32K token）              │
│                                                     │
│  系统提示    用户历史对话    检索到的文档    当前问题   │
│  (~500)      (~2000)        (~20000)       (~200)   │
│                                                     │
└─────────────────────────────────────────────────────┘
                                              ↓
                                         模型生成回答
```

窗口内的所有内容，模型都能"看到"并参考。窗口之外的内容，模型完全不知道。

## 为什么上下文窗口对 RAG 至关重要

### 更大窗口 = 更多参考信息

RAG 的核心思路是"检索相关文档，让模型基于文档回答"。上下文窗口越大：

- 可以放入更多检索结果，覆盖更复杂的问题
- 可以保留更长的对话历史，支持多轮追问
- 可以放入完整的大文件，而非截断的片段

### 但更大不总是更好

研究表明，当上下文过长时，模型的注意力会分散，对中间部分的信息利用率下降——这被称为**"Lost in the Middle"问题**（详见后文）。

## 上下文窗口的演进

```
2020  GPT-3          2,048 token   (~1,500 词)
2022  GPT-3.5        4,096 token   (~3,000 词)
2023  Claude 2      100,000 token  (~75,000 词)
2023  GPT-4         128,000 token  (~96,000 词)
2024  Gemini 1.5  1,000,000 token  (~750,000 词)  ← 约 750 本小说
2024  Qwen2.5       128,000 token  本地可部署
2025  Qwen2.5-1M  1,000,000 token  本地可部署（需高端硬件）
```

上下文窗口的快速扩展，得益于以下技术突破：

### RoPE（旋转位置编码）

传统 Transformer 使用绝对位置编码，难以外推到训练时未见过的长度。RoPE 通过旋转矩阵编码相对位置，具有更好的长度泛化能力：

```
传统位置编码: position_embedding[0], [1], [2], ..., [2047]
             超出 2048 就没有对应的编码了

RoPE: 通过旋转角度编码相对距离
      position i 和 position j 的关系只取决于 (i-j)
      理论上可以外推到任意长度
```

### YaRN（Yet another RoPE extensioN）

YaRN 是 RoPE 的改进版，通过调整旋转频率，让模型能在不重新训练的情况下处理更长的上下文：

```python
# YaRN 的核心思想：对不同频率的位置编码分别缩放
# 低频分量（捕捉长距离依赖）：线性外推
# 高频分量（捕捉局部模式）：NTK 插值
```

Qwen2.5 使用 YaRN 将基础的 8K 训练长度扩展到 128K，同时保持短文本的性能。

## Lost in the Middle：有效上下文 vs 名义上下文

2023 年的研究论文《Lost in the Middle》揭示了一个重要现象：**模型对上下文开头和结尾的信息利用率，显著高于中间部分。**

```
上下文位置对模型注意力的影响（示意）：

注意力
  ↑
高 │█                                              █
   │ █                                            █
   │  █                                          █
   │   ██                                      ██
低 │     ████████████████████████████████████
   └────────────────────────────────────────────→
   开头                                        结尾
                    上下文位置
```

**实际影响：**

```
检索到 10 个文档，正确答案在第 5 个文档中：
- 放在位置 1（开头）：模型正确率 ~80%
- 放在位置 5（中间）：模型正确率 ~55%
- 放在位置 10（结尾）：模型正确率 ~75%
```

这意味着，即使上下文窗口足够大，放入过多文档也可能降低回答质量。

## 上下文管理策略

### 策略一：Stuff（直接填充）

将所有检索结果直接放入上下文，一次性生成回答。

```
[系统提示] + [文档1] + [文档2] + ... + [文档N] + [用户问题] → 回答
```

**适用场景：** 文档总量在上下文窗口内，且文档数量不超过 5~8 个。

**优点：** 简单，模型能看到所有信息，可以综合推理。

**缺点：** 文档过多时，Lost in the Middle 问题严重。

### 策略二：Map-Reduce（分而治之）

将文档分批处理，再汇总结果：

```
第一阶段（Map）：
  [文档1] + [问题] → 部分回答1
  [文档2] + [问题] → 部分回答2
  [文档3] + [问题] → 部分回答3

第二阶段（Reduce）：
  [部分回答1] + [部分回答2] + [部分回答3] + [问题] → 最终回答
```

**适用场景：** 文档总量超出上下文窗口，或需要处理大量文档。

**优点：** 可处理任意数量的文档。

**缺点：** 多次 LLM 调用，延迟高；跨文档的关联信息可能丢失。

### 策略三：Refine（迭代精炼）

逐步将文档信息融入回答：

```
初始回答 = LLM([文档1] + [问题])
精炼回答1 = LLM([文档2] + [初始回答] + [问题])
精炼回答2 = LLM([文档3] + [精炼回答1] + [问题])
...
最终回答 = 精炼回答N
```

**适用场景：** 需要综合多个文档，且文档有顺序关系。

**优点：** 每步都能利用之前的推理结果。

**缺点：** 串行处理，延迟最高；早期错误会传播。

## Delphi 的上下文预算管理

Delphi 采用**动态上下文预算**策略，在回答质量和响应速度之间取得平衡。

### 预算分配

```
总上下文窗口: 32,768 token（默认配置）
├── 系统提示:     500 token（固定）
├── 对话历史:   2,000 token（最近 N 轮）
├── 检索文档:  28,768 token（动态分配）
└── 回答预留:   1,500 token（生成空间）
```

### 文档排序优化

针对 Lost in the Middle 问题，Delphi 对检索结果重新排序后再放入上下文：

```python
# 将最相关的文档放在开头和结尾，次相关的放在中间
# 示例：5 个文档，相关性排序为 [A, B, C, D, E]
# 重排后放入上下文：[A, C, E, D, B]
#                   ↑最相关  ↑次相关  ↑最相关（结尾）
```

### 配置示例

```yaml
# config.yaml
context:
  window_size: 32768        # 总上下文窗口
  system_prompt_budget: 500
  history_budget: 2000
  answer_budget: 1500
  # 检索文档预算 = window_size - 以上三项

  rerank: true              # 启用重排序，缓解 Lost in the Middle
  max_chunks: 10            # 最多放入 10 个 chunk
  strategy: stuff           # stuff | map_reduce | refine
```

### 自动策略选择

```
检索到的文档总 token 数
        ↓
  ≤ 可用预算？
  ├── 是 → 使用 Stuff 策略（最快，质量最好）
  └── 否 → 文档数 ≤ 20？
            ├── 是 → 截断到预算内，保留最相关的文档
            └── 否 → 使用 Map-Reduce 策略
```

# BGE-M3 嵌入模型

## 什么是 BGE-M3

**BGE-M3**（BAAI General Embedding - Multi-lingual, Multi-functionality, Multi-granularity）是由北京智源人工智能研究院（BAAI）开源的嵌入模型。它在 2024 年发布，是目前综合能力最强的开源嵌入模型之一。

"M3" 代表三个核心维度：

- **Multi-lingual（多语言）**：支持 100+ 种语言，中英文效果尤为出色
- **Multi-functionality（多功能）**：同时支持稠密检索、稀疏检索和多向量检索
- **Multi-granularity（多粒度）**：支持从短句到长达 8192 tokens 的长文档

## 三种输出模式

BGE-M3 最独特的特性是能够同时输出三种不同类型的向量表示，每种都有其适用场景。

### Dense（稠密向量）

传统的语义嵌入，将文本映射为一个固定维度（1024 维）的稠密浮点向量：

```
"如何实现快速排序" → [0.023, -0.156, 0.891, ..., 0.034]  # 1024 维
```

- 擅长捕捉语义相似性
- 适合跨语言检索（中文问题匹配英文文档）
- 使用余弦相似度或内积计算相关性

### Sparse（稀疏向量）

类似传统 BM25 的词频权重，但由神经网络学习得到，输出为词汇表上的稀疏权重分布：

```
"快速排序算法" → {
  "快速": 0.82,
  "排序": 0.91,
  "算法": 0.67,
  # 其余 30000+ 词汇权重为 0
}
```

- 擅长精确关键词匹配
- 对专有名词、函数名、变量名等效果好
- 可解释性强，便于调试

### ColBERT（多向量表示）

为文本中的每个 token 生成独立的向量，通过"最大相似度求和"（MaxSim）计算相关性：

```
"快速排序" → [v_快, v_速, v_排, v_序]  # 每个 token 一个向量

相关性 = Σ max(query_token_i · doc_token_j)
```

- 细粒度的词级匹配，精度最高
- 存储和计算开销较大
- 适合需要高精度的重排序（Reranking）场景

### 三种模式的对比

```
┌──────────────┬──────────────┬──────────────┬──────────────┐
│              │    Dense     │    Sparse    │   ColBERT    │
├──────────────┼──────────────┼──────────────┼──────────────┤
│ 向量维度     │ 1024 维      │ ~30K 稀疏    │ N × 128 维   │
│ 语义理解     │ ★★★★★       │ ★★★          │ ★★★★         │
│ 关键词匹配   │ ★★★          │ ★★★★★       │ ★★★★         │
│ 存储开销     │ 低           │ 低（稀疏）   │ 高           │
│ 检索速度     │ 快           │ 快           │ 慢           │
│ 典型用途     │ 语义检索     │ 关键词检索   │ 精排/重排序  │
└──────────────┴──────────────┴──────────────┴──────────────┘
```

## 为什么多输出对代码搜索至关重要

代码搜索是一个特殊的检索场景，同时需要语义理解和精确匹配：

```
用户查询："怎么用 asyncio 实现并发请求"

Dense 的优势：
  · 理解"并发请求"的语义，匹配到使用 aiohttp、httpx 的代码
  · 即使代码注释是英文也能匹配

Sparse 的优势：
  · 精确匹配 "asyncio" 这个关键词
  · 不会把同步的 requests 库代码误匹配进来

ColBERT 的优势：
  · 在候选结果中精确区分 asyncio.gather 和 asyncio.wait 的细微差别
```

**混合检索（Hybrid Search）** 结合 Dense 和 Sparse 的结果，再用 ColBERT 重排序，是目前代码搜索的最佳实践。

## 模型架构与训练

BGE-M3 基于 **XLM-RoBERTa** 架构，参数量约 570M。训练过程分为三个阶段：

```
阶段 1：RetroMAE 预训练
  · 在大规模多语言语料上进行掩码自编码预训练
  · 建立基础的多语言语义理解能力

阶段 2：多任务微调
  · 同时优化 Dense、Sparse、ColBERT 三个目标
  · 使用对比学习（Contrastive Learning）
  · 训练数据包含多语言问答对、代码-注释对等

阶段 3：自蒸馏
  · 用三种模式的集成结果作为软标签
  · 进一步提升各模式的单独性能
```

最大支持序列长度为 **8192 tokens**，远超大多数嵌入模型的 512 tokens 限制，使其能够处理完整的代码文件或长篇文档。

## 与其他嵌入模型的对比

| 模型 | 维度 | 最大长度 | 多语言 | 开源 | 多输出 |
|------|------|---------|--------|------|--------|
| BGE-M3 | 1024 | 8192 | ✓ (100+) | ✓ | ✓ (3种) |
| OpenAI text-embedding-3-large | 3072 | 8191 | ✓ | ✗ | ✗ |
| Jina-embeddings-v3 | 1024 | 8192 | ✓ | ✓ | ✗ |
| E5-mistral-7b | 4096 | 32768 | 有限 | ✓ | ✗ |
| Cohere embed-v3 | 1024 | 512 | ✓ | ✗ | ✗ |

BGE-M3 的核心竞争力在于：**完全开源、本地部署、多输出模式**，这对于隐私敏感的企业知识库场景尤为重要。

## 基准测试表现

在 MTEB（Massive Text Embedding Benchmark）多语言榜单上，BGE-M3 在中文检索任务中表现突出：

| 任务类型 | BGE-M3 | OpenAI-3-large | E5-large |
|---------|--------|----------------|---------|
| 中文语义检索 | **72.3** | 68.1 | 61.4 |
| 中英跨语言检索 | **68.9** | 65.2 | 58.7 |
| 代码检索 | **67.4** | 64.8 | 55.2 |
| 长文档检索 | **71.1** | 69.3 | 60.8 |

> 数据来源：MTEB Leaderboard，分数为 nDCG@10

## Delphi 如何使用 BGE-M3

Delphi 充分利用了 BGE-M3 的多输出特性，构建了混合检索流水线：

```
文档导入阶段：
┌─────────────────────────────────────────────┐
│  文本分块（Chunking）                        │
│       │                                     │
│       ▼                                     │
│  BGE-M3 编码                                │
│  ├── Dense 向量 → Qdrant（ANN 索引）          │
│  ├── Sparse 向量 → Qdrant（稀疏索引）         │
│  └── 元数据 → Qdrant（Payload 存储）          │
└─────────────────────────────────────────────┘

查询阶段：
┌─────────────────────────────────────────────┐
│  用户查询                                    │
│       │                                     │
│       ▼                                     │
│  BGE-M3 编码查询                            │
│  ├── Dense 检索 → Top-K 候选                │
│  └── Sparse 检索 → Top-K 候选               │
│       │                                     │
│       ▼                                     │
│  RRF 融合（Reciprocal Rank Fusion）          │
│       │                                     │
│       ▼                                     │
│  BGE-Reranker 精排（可选）                   │
│       │                                     │
│       ▼                                     │
│  Top-N 结果 → 注入 LLM Prompt               │
└─────────────────────────────────────────────┘
```

通过混合检索，Delphi 在代码库问答场景中同时保证了语义理解（"这个函数是做什么的"）和精确匹配（"找到所有调用 `connect_db` 的地方"）的检索质量。

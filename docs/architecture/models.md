# 模型选型

## LLM 基座模型

### 默认模型：Qwen3.5-27B

| 属性 | 值 |
|------|-----|
| 参数量 | 27B |
| 显存需求 | ~20GB |
| 上下文长度 | 128K tokens |
| 架构 | Dense Transformer |

选择理由：
- 代码理解基准测试中表现接近 GPT-4 水平
- 单卡 RTX 4090 (24GB) 可运行
- 128K 上下文足以容纳长调用链 + 检索上下文
- 中英文双语能力优秀，适合中文技术文档 + 英文代码混合场景

### 备选模型

| 模型 | 参数量 | 显存 (Q4) | 适用场景 |
|------|--------|----------|---------|
| DeepSeek-Coder-V2-Lite-Instruct | 16B (2.4B active) | ~10GB | MoE 架构，推理速度快 |
| Qwen2.5-Coder-14B-Instruct | 14B | ~10GB | 平衡性能与速度 |
| Qwen2.5-Coder-7B-Instruct | 7B | ~5GB | 低显存环境，速度最快 |

用户可替换为任意 HuggingFace 格式或 GGUF 格式的模型。

## Embedding 模型

### 默认模型：BGE-M3 (BAAI)

| 属性 | 值 |
|------|-----|
| 输出维度 | 1024 |
| 最大输入 | 8192 tokens |
| 模型大小 | ~2.2GB |
| 特殊能力 | Dense + Sparse 双向量输出 |

选择理由：
- 中英文混合场景下表现最优
- 原生支持稠密向量（语义匹配）+ 稀疏向量（关键词匹配）双输出
- 8192 tokens 输入长度足以容纳较长的函数体和文档段落
- 可在 CPU 上运行（速度较慢），也可用 GPU 加速

### Dense + Sparse 双向量的意义

纯语义检索在代码场景下有明显缺陷。例如查询 `std::unique_ptr<apollo::cyber::Node>` 时，语义模型可能关注"智能指针"的宏观概念而忽略具体的类型名。

BGE-M3 的 Sparse 输出类似 BM25 的词频匹配，能精确捕获这类符号级关键词。两者结合实现"既理解意图，又不漏关键词"。

## Reranker 模型

### 默认模型：BGE-Reranker-v2-M3

| 属性 | 值 |
|------|-----|
| 架构 | Cross-encoder |
| 参数量 | ~560M |
| 运行环境 | CPU 可跑，GPU 更快 |

选择理由：
- 交叉编码器架构，将 Query 和 Document 联合输入做深层注意力计算
- 精排效果显著优于纯向量相似度
- 模型小，不占用 LLM 的 GPU 显存，可跑在 CPU 上
- 与 BGE-M3 同系列，兼容性好

### Reranker 的作用

向量检索的 Top-20 结果中可能包含"变量名相似但逻辑无关"的噪音 Chunk。Reranker 通过深层语义比对，将真正相关的 Chunk 排到前面，通常将 Top-20 精排为 Top-5 送入 LLM。

## 音视频转录模型

### 默认模型：Faster-Whisper large-v3

| 属性 | 值 |
|------|-----|
| 基础模型 | OpenAI Whisper large-v3 |
| 加速框架 | CTranslate2 |
| 量化 | INT8 |
| 速度 | 约 4× 原生 Whisper |

选择理由：
- INT8 量化后速度提升约 4 倍，内存占用大幅降低
- 词错误率（WER）与原生模型基本一致
- 支持中英文自动检测
- 单卡 4090 可实时转录
